# Reproducing the NDVM Results

This document is the artifact-level reproducibility guide for the NDVM paper. It identifies the
public code channel, the exact machine and software the reported numbers were taken on, the one
command that captures that environment, a figure/table to script map, the run protocol, and the
tolerances, seeds, and run counts behind every numeric gate. It also records the measurement
caveats so that a reviewer can judge what each number does and does not establish.

Every number in the paper comes from a real run on the hardware described below. No figure value
is hand-edited.


## 1. Public artifact channel

- Repository: `github.com/sheneman/ndvm`
- Frozen tag for this paper: `v1.0` (checkout with `git checkout v1.0`). The default branch stays
  live; the tag is the stable, citable tree behind the reported numbers.
- The exact commit that produced a given run is recorded inside that run's own outputs and in the
  environment manifest (see Section 2). To read the commit of a working tree:

  ```
  git rev-parse HEAD
  ```

The native runtime, the PyTorch DMCI oracle it is validated against, the profiling and baseline
harnesses, the differential-testing suite, and the second-client demonstrations all live under
`ndvm/`, with the oracle package under `neural_compiler/` and the Scheme evaluator source under
`bootstrap/`. The manuscript is distributed separately.


## 2. Hardware and software the numbers were taken on

All timing in the paper was taken on a single compute node, **n128**, of the `sheneman` partition
of a university cluster. The complete environment is captured by one command, which must be run on
the same node as the measurement (a compute node, not the login node):

```
NDVM_PY=.venv/bin/python bash ndvm/profiling/env_manifest.sh env_manifest.txt
```

`env_manifest.sh` emits commit, host, CPU model/cores/cache, NUMA topology, memory, governor,
compiler versions and build flags, NumPy/BLAS and PyTorch versions and thread counts, OS, and
`perf` access state. Treat this script as the canonical, one-command environment capture; cite the
manifest it produces alongside any runtime figure.

The manifest captured on n128 for this paper reads (abridged):

| Field | Value |
| --- | --- |
| Host | `n128.fortytwo.ibest.uidaho.edu` (partition `sheneman`) |
| CPU | AMD Ryzen Threadripper PRO 5975WX, 32 cores / 64 threads, 1 socket, 1 NUMA node |
| Caches | L1d/L1i 32K, L2 512K, L3 32768K; SIMD up to AVX2/FMA |
| Frequency governor | `performance` |
| Memory | 125 GiB total |
| Compiler | GCC 12.1.0 (`module load gcc/12.1.0`); CMake `Release` = `-O3` (see `ndvm/CMakeLists.txt`); Python extension build via `ndvm/python/setup.py` |
| Python stack | PyTorch 2.12.0, NumPy 2.4.6, `torch.get_num_threads()` = 32 |
| OS | Rocky Linux 8.10 (Green Obsidian), kernel 4.18.0-553 x86_64 |
| `perf` access | `perf_event_paranoid = 2`, `perf` binary not present (see Section 6) |

The native runtime is also validated under both `clang++` and `g++`; the g++ build is the one used
on the cluster. Validate on the deployment compiler: g++ surfaced a forward-pass evaluation-order
issue that the clang/ASan build did not, so equivalence is checked on the same toolchain that runs
the benchmarks.

## 4. Run protocol

The `*.sbatch` files and the `srun` commands below are verbatim records of the cluster protocol
behind the paper's numbers (Slurm partition and node paths included); on your own machine, run the
same Python scripts directly. On the original cluster: the login node lacks PyTorch; the repository
`.venv` resolves correctly only on a compute node. Run every measurement through `srun` on the
`sheneman` partition. The pattern for one script:

```
module load gcc/12.1.0
srun --partition=sheneman --time=20:00 --cpus-per-task=4 --mem=12G \
     .venv/bin/python ndvm/profiling/profile_dmci_baseline.py --iters 50 --batches 1 8 64 256 --decompose
```

The other scripts follow the same `srun ... .venv/bin/python <script> [args]` form:

- `ndvm/profiling/alloc_counters.py`
- `ndvm/profiling/residual_e2e.py` (program names as positional args, e.g. `kalman2d_T80`, `slow`; rep count via `NDVM_REPS`, default 15)
- `ndvm/profiling/jax_baseline.py --iters 50 --batch 256`
- `ndvm/profiling/staged_baseline.py --iters 30 --curve-max 2000`
- `ndvm/tests/run_fuzz.py --n 200 --seed 1234` (and `--n 500 --seed 7`)
- `ndvm/profiling/bytecode_vm_e2e.py`
- `ndvm/profiling/cosearch_budget.py`

Build the native extension once on the compute node before the first run (it is node-local; build
and run inside the same `srun` allocation if `/tmp` is node-local). Capture the environment manifest
(Section 2) inside the same allocation so the commit and host are recorded next to the numbers.


## 5. Tolerances, seeds, and number of runs for the gates

Equivalence and correctness gates are float32, since the DMCI oracle and NDVM are float32-native.

Differential tester (`ndvm/tests/run_fuzz.py`), three gates per program:

| Gate | Comparison | atol | rtol |
| --- | --- | --- | --- |
| G1 forward | NDVM forward vs DMCI oracle forward | 1e-4 | 1e-4 |
| G2 gradient | NDVM reverse-mode d/dp vs oracle autograd d/dp | 2e-3 | 2e-3 |
| G3 finite-diff | NDVM d/dp vs central finite difference of the NDVM forward | 1e-2 | 5e-2 |

- Central finite-difference step `FD_EPS = 1e-3`.
- Seeds and corpus sizes: the canonical corpus is `N = 200` programs at seed `1234` (600 checks,
  3 per program); an independent corpus is `N = 500` programs at seed `7` (1500 checks). The
  generator is deterministic given a seed and freezes its corpus to disk so the same programs are
  re-checked on every run.
- G2 cross-validates the native tape against an independent reverse-mode engine (the oracle); G3
  self-validates the tape against the runtime's own forward.

Timing runs use medians over repeated iterations (`--iters` for the profilers, `NDVM_REPS` for the
3-backend decomposition). Forward equivalence and the Kalman end-to-end NLL trajectory are checked
to float32 as above; the calibration driven by NDVM follows the same Adam trajectory as the PyTorch
backend because the gradients are exact, so the comparison is at equal quality.


## 6. Measurement caveats

These are the honest limits on what each instrument establishes.

- **cProfile time is distortable.** `cProfile` adds per-call overhead and over-weights
  small-call-heavy programs. We therefore use it only for *bucket shares* (which class of work
  dominates the forward pass), never for absolute timing claims.
- **Allocation counts are the bias-free cross-check.** The cost claim rests on directly counting
  tagged-value constructors and peak transient memory (`alloc_counters.py`), not on profiled time.
  This is the instrument the cost argument should be read against.
- **Hardware performance counters are unavailable on the cluster.** `perf_event_paranoid = 2` and
  no `perf` binary on the compute node, so cache-miss / instruction counters cannot be collected.
  Allocation counts are the reported evidence in their place.
- **Partition choice.** The `eight` partition is noisy; all reported timing is taken on n128 of the
  `sheneman` partition under the `performance` governor.
- **Compiler.** Validate equivalence on the deployment compiler (g++ 12.1.0 here); a g++-only
  forward evaluation-order issue did not reproduce under clang/ASan.
